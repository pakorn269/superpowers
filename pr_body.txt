## What problem are you trying to solve?
The `brainstorm-server` performs an asynchronous disk read (`fs.promises.readFile`) and executes heavy string manipulations (`lastIndexOf` and `slice` for injecting scripts) on the root `GET /` route for *every single incoming HTTP request*. Under load or frequent reloading, reading multi-megabyte HTML screens from disk and performing string manipulation on the event loop creates a severe performance bottleneck.

## What does this PR change?
It implements an in-memory cache (`cachedHtmlResponse`) that pre-constructs and caches the complete HTML response string exactly once per file update, allowing the `GET /` route to serve the response immediately from memory without executing disk I/O.

## Is this change appropriate for the core library?
Yes, this improves the core performance and responsiveness of the brainstorm-server, which is fundamental infrastructure for the visual-brainstorming workflows.

## What alternatives did you consider?
I considered switching to synchronous `fs.readFileSync` to avoid async promise overhead, but this would still perform disk I/O and block the event loop on every request. I also considered caching just the raw file content, but that would still require executing the string manipulation per request. Caching the fully constructed response is the most efficient approach.

## Does this PR contain multiple unrelated changes?
No, it strictly focuses on implementing the HTML string cache for the `brainstorm-server` root route.
