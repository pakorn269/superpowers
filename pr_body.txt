## What problem are you trying to solve?
The `/files/` endpoint in `skills/brainstorming/scripts/server.cjs` extracts the requested file name directly from `req.url` without decoding it. When a request is made for a file with encoded characters (e.g., spaces represented as `%20`), the server looks up the literal encoded string instead of the decoded name. This fails to serve the intended file and could potentially lead to unexpected path lookup behavior. Since `req.url` in Node.js native `http` is not automatically URL-decoded, it must be handled explicitly. Additionally, calling `decodeURIComponent` on malformed URIs (e.g. `%FF`) can throw an exception, crashing the server.

## What does this PR change?
This PR modifies the `/files/` endpoint to safely decode the URI component of the requested file name before joining it with `CONTENT_DIR`. It wraps the decoding step in a `try/catch` block to handle malformed URIs safely, returning a `400 Bad Request` instead of crashing the server.

## Is this change appropriate for the core library?
Yes, this is a bug fix for the core brainstorming skill server, ensuring files with encoded characters are served correctly and preventing a potential Denial of Service (DoS) from unhandled URI exceptions.

## What alternatives did you consider?
Using an external routing library (e.g., Express) was considered, but it would add unnecessary dependencies for a single endpoint. Explicitly decoding the URI with native Node.js functionality is the most lightweight and correct approach.

## Does this PR contain multiple unrelated changes?
No, this PR only fixes the URL decoding issue.
